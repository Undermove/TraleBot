# OWNER-PRIORITIES

Это источник правды по задачам, которые **владелец** (Undermove) явно поставил в первый приоритет. Любой агент в начале своей сессии **обязан прочитать этот файл первым** и выполнить свою часть по каждой задаче **до** обычной работы (аудит, квоты, новые идеи, breakdown мелочёвки).

Файл живёт в корне репозитория. Только Product-агент имеет право добавлять/удалять задачи отсюда (по итогам триажа owner-driven запросов). Остальные агенты только отмечают свои пункты как выполненные.

---

## Как читать

Каждая задача — секция `## §<номер из ROADMAP> «<название>»` со следующей структурой:

- **Pinned** — дата, когда владелец поставил задачу в приоритет
- **ROADMAP-статус** — текущий статус в ROADMAP.md (`[idea]` / `[designed]` / `[dev]` / `[review]`)
- **Чек-лист по ролям** — что нужно от каждого агента, чтобы задача дошла до зелёных тестов в проде
- **Пометки** — что уже сделано

Когда пункт чек-листа закрыт, агент ставит `[x]` и добавляет короткий комментарий: `→ done by <agent> <date>: <что сделано / куда положил>`.

Когда **все** пункты задачи закрыты, Product-агент удаляет её из этого файла на следующей сессии.

---

## Правила приоритетов

1. **Owner-priority бьёт всё.** Пока в этом файле есть незакрытые пункты для твоей роли — обычная работа сессии (аудит, генерация идей, выбор `[idea]` из ROADMAP) **запрещена**. Сначала закрываешь свои пункты здесь.

2. **Свои квоты не применяются.** Если у тебя стоит квота «не больше N issues за сессию» — она про обычную работу. Owner-priority не считается в эту квоту.

3. **Если пункт неприменим** — например, спека уже написана и не нужна вторая — отметь `[x]` с пометкой `→ already done in <commit/file>`. Не пропускай молча.

4. **Если пункт упёрся в блокер** — например, требует решения, которое выше твоего уровня (продуктовое, инфраструктурное) — НЕ ставь `[x]`, добавь под пунктом строку `⚠ blocked: <причина>`. Product следующей сессии разберётся.

5. **Не плоди дубликатов.** Если открываешь GitHub issue из owner-priority задачи — указывай в title и body, что это часть owner-priority §N.

## Активные owner-priorities

### §80 «Конструктор предложений — расширение на 3 модуля»

- **Pinned:** 2026-05-08 (после успешного приёмочного теста §79 на Postpositions L7)
- **ROADMAP-статус:** `[idea]`
- **Описание:** §79 запустил механику SentenceBuilder только на Postpositions. Расширить тип упражнения на ещё три модуля, где синтаксис критичен: **Cases** (падежи — закрепить эргатив/датив через сборку), **Present Tense** (базовый SOV), **Cafe / Shopping / Taxi** (живые туристические фразы как полные предложения). Реиспользуем существующий `SentenceBuilderCard` без правок UI.
- **Целевые модули:** cases, present-tense, cafe, shopping, taxi (пять модулей; cafe/shopping/taxi считаются группой тематических, у каждого по одному lesson)

**Чек-лист по ролям:**

- [ ] **methodist** — для каждого из 5 модулей подобрать 5–7 русско-грузинских пар × 3 уровня (L1: один пустой слот / L2: два слота / L3: половина предложения). Сохранить в `design-specs/80-sentence-builder-spread-content.md`. Cases — фокус на эргативе и дативе. Present Tense — SOV с глаголами Класса 1 и 2 (читать, писать, видеть, идти). Cafe/Shopping/Taxi — по одному ключевому предложению на модуль («Я хочу кофе», «Сколько стоит этот чай», «Поехали в Батуми»).
- [ ] **native-reviewer** — для каждого предложения проверить единственный корректный порядок слов, дистракторы реально сбивают, регистр естественный (нейтральный разговорный, без книжных архаизмов).
- [ ] **tech-lead-breakdown** — разбить на task-issues по одному lesson на модуль:
  1. Cases lesson N+1 sentence-builder + JSON + theory-блок + ModuleRegistry MaxLessons +1.
  2. Present Tense lesson N+1 — то же.
  3. Cafe lesson N+1.
  4. Shopping lesson N+1.
  5. Taxi lesson N+1.
  6. Расширение `SentenceBuilderContentValidationTests` для покрытия новых модулей.
- [ ] **developer** — реализовать в порядке tech-lead'а. Каждая таска = один lesson + JSON + theory + регистрация в ModuleRegistry. UI не трогать — `SentenceBuilderCard` и сабкомпоненты уже на месте.
- [ ] **qa** — после landing'а каждого PR пройти full-suite (1430+ dotnet, 13+ Playwright, 4+ vitest) + ручная проверка одного happy-path сценария на каждый модуль через `@traletest_bot` на 375px.
- [ ] **product** — после полной готовности (все 5 lesson'ов в проде, тесты зелёные) убрать §80 из этого файла и обновить ROADMAP §80 в `[done]`.

## Как владелец добавляет новый owner-priority

Через комментарий в любой sprint-plan-issue: «приоритезируй §N как owner-priority». Product следующей сессии:
1. Перечитывает описание §N из ROADMAP.
2. Заводит секцию в OWNER-PRIORITIES.md по шаблону выше с чек-листом по ролям.
3. Если §N ещё не существует в ROADMAP — создаёт `[idea]` и сразу пиннит.

Если задача требует ролей, не указанных в чек-листе по умолчанию (например, deploy, marketing) — Product добавляет нужные роли.
