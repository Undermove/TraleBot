## Закрытые таски

### #950 — Domain: NotificationTrigger + NotificationsEnabled + /notifications opt-out

**Что проверяем:** команда бота `/notifications off/on` позволяет юзеру управлять подпиской на пуш-уведомления; флаг сохраняется в базе и будет учитываться воркером пушей.

**Шаги — отключение уведомлений:**

1. Открой @trale_bot (или @traletest_bot) в Telegram.
2. В чате с ботом отправь сообщение: `/notifications off`
3. Бот должен ответить текстом «Уведомления отключены. Чтобы включить снова — /notifications on».
4. Никаких кнопок, никаких ошибок — только текстовый ответ.

**Шаги — повторное включение уведомлений:**

5. Отправь в тот же чат: `/notifications on`
6. Бот должен ответить: «Уведомления включены. Чтобы отключить — /notifications off».

**Шаги — неизвестный аргумент:**

7. Отправь: `/notifications maybe`
8. Бот должен ответить: «Используй /notifications on или /notifications off».

**Шаги — проверка идемпотентности:**

9. Если уведомления уже отключены (шаг 2 выполнен), снова отправь `/notifications off`.
10. Бот должен снова ответить «Уведомления отключены...» — без ошибки, без исключения.

**Ожидаемое поведение:** на все три вариации команды бот отвечает корректным текстом без ошибок. Флаг сохраняется мгновенно — при повторной команде `/notifications on` бот включает уведомления обратно.

**Если что-то не так:**
- Бот не отвечает → F12 → Network, найди POST `/api/...` от бота, проверь 500-ошибку; проверь логи `dotnet run` на предмет исключений `NotificationsCommand`.
- Ответ не тот текст → проверь, точный ли текст команды (пробел между `/notifications` и `off`).
- Команда вообще не распознаётся (бот отвечает не тем) → проверь регистрацию `NotificationsCommand` в `DependencyInjection.cs` (grep по `NotificationsCommand`).

### #951 — Application: DailyReturnNotificationService — eligibility, cooldown, отправка пуша

**Что проверяем:** сервис автоматически отправляет пуш-сообщение в Telegram юзерам, которые начали урок, но не заходили больше суток; соблюдает 7-дневный кулдаун и уважает флаг NotificationsEnabled.

> Это backend-сервис без отдельного UI. Пуш отправляется один раз в сутки в 10:00 UTC автоматически через фоновый воркер `ReturnPushWorker`. Ниже — как убедиться, что инфраструктура работает, и как дождаться реального пуша.

**Шаги — проверка структуры базы данных (локально через Swagger):**

1. Открой http://localhost:1402/swagger в браузере.
2. Найди секцию Admin → `GET /api/admin/users/{telegramId}`.
3. Введи свой Telegram ID (можно узнать у @userinfobot) и нажми Execute.
4. В ответе JSON проверь, что поле `notificationsEnabled` присутствует (значение `true` или `false`).

**Шаги — проверка что пуш не пойдёт юзеру с отключёнными уведомлениями:**

5. В чате с ботом отправь `/notifications off` (см. тест #950 выше).
6. Дождись следующего срабатывания воркера (10:00 UTC = 14:00 по Батуми) или проверь логи запущенного сервера.
7. В логах сервера (консоль `dotnet run`) НЕ должно быть строки с SendDailyReturnPushAsync для твоего userId.

**Шаги — проверка что пуш приходит eligible юзеру (требует тестового аккаунта):**

> Для этого теста нужен аккаунт, у которого в `MiniAppUserProgresses` есть `CompletedLessonsJson != {}` (прошёл хотя бы один шаг урока) и `LastPlayedAtUtc` старше 30 часов.

8. Убедись, что в тестовом аккаунте флаг `NotificationsEnabled = true` (отправь `/notifications on`).
9. В мини-аппе (@traletest_bot) пройди хотя бы один шаг любого урока — например, **Главный экран → плитка «Алфавит» → Урок 1 → один вопрос**.
10. Не заходи в мини-апп минимум 30 часов.
11. На следующий день около 14:00 по Батуми (10:00 UTC) в чат с ботом должно прийти сообщение вида «Бомбора по тебе скучает 🐶 Продолжишь урок сегодня?» или «Твой урок ждёт продолжения 📖 Вернёшься?» с кнопкой «▶️ Продолжить».
12. Проверь: кнопка «▶️ Продолжить» должна быть кнопкой типа WebApp (при тапе открывает мини-апп, не внешнюю ссылку).

**Шаги — проверка кулдауна (через логи):**

13. После получения первого пуша (шаг 11) не жди 7 дней — вместо этого проверь логи следующего запуска воркера в 10:00 UTC следующего дня.
14. В логах сервера должна быть строка вида «skipped due to cooldown» для твоего userId (повторный пуш не отправляется пока не прошло 7 дней).

**Ожидаемое поведение:** при соответствующих условиях в 10:00 UTC бот присылает пуш с именем модуля и кнопкой, ведущей прямо в урок. Повторные пуши заблокированы на 7 дней. Юзеры с `NotificationsEnabled = false` пуши не получают.

**Если что-то не так:**
- Пуш не пришёл в нужное время → проверь логи `ReturnPushWorker started` / `ReturnPushWorker iteration failed`. Проверь, что `LastPlayedAtUtc` в базе действительно старше 30 часов.
- Пуш пришёл, но текст без имени модуля → в логах ищи ошибки десериализации `CompletedLessonsJson`.
- Пуш пришёл дважды в один день → проверь, нет ли дублирующей записи в таблице `NotificationTriggers` с `source = 'daily_return'` для твоего telegramId; одна запись должна обновляться (upsert), а не создаваться заново.

### #953 — Mini-app: deep-link routing для push CTA (moduleId + lessonId)

**Что проверяем:** когда мини-апп открывается по ссылке с параметрами `?moduleId=X&lessonId=Y` (например, из пуш-сообщения), он сразу переходит на нужный экран урока — без вспышки главного экрана; кнопка «Назад» возвращает на главный экран, а не в модуль; при невалидных параметрах показывается главный экран без ошибок.

**Шаги — прямой переход по deep-link:**

1. Узнай ID одного из модулей. Например, модуль «Алфавит» имеет id `alphabet`. В мини-аппе пройди хотя бы 1–2 урока в этом модуле (чтобы урок был в прогрессе).
2. В браузере на телефоне (или в Telegram через ссылку) открой мини-апп с параметрами: `https://<твой-ngrok-домен>/?moduleId=alphabet&lessonId=3` (замени `3` на номер урока, который есть в твоём прогрессе).
3. Мини-апп должен открыться **сразу на экране теории или практики урока 3 модуля «Алфавит»** — без промежуточного показа главного дашборда.
4. Убедись, что вспышки главного экрана не было (экран урока появляется первым).

**Шаги — кнопка «Назад» из deep-linked урока:**

5. Находясь на экране урока из шага 3, нажми системную кнопку «Назад» Telegram (стрелка влево вверху).
6. Должен открыться **главный экран (Dashboard)** — список модулей, а не экран модуля «Алфавит».
7. Ещё раз тапни «Назад» — мини-апп должен закрыться (стандартное поведение Telegram WebApp).

**Шаги — fallback при невалидных параметрах:**

8. Открой URL с несуществующим модулем: `https://<твой-ngrok-домен>/?moduleId=nonexistent&lessonId=99`
9. Должен открыться **главный экран (Dashboard)** — без ошибок, без белого экрана, без «Cannot read properties of undefined».
10. В консоли браузера (F12) не должно быть непойманных JS-ошибок.

**Шаги — fallback при частичных параметрах:**

11. Открой URL только с одним параметром: `https://<твой-ngrok-домен>/?moduleId=alphabet` (без lessonId).
12. Должен открыться главный экран (Dashboard) — не экран урока.

**Шаги — проверка события push_clicked:**

13. В браузере открой F12 → вкладка Network.
14. Открой мини-апп с валидными параметрами (шаг 2).
15. В Network должен пройти POST-запрос на `/api/miniapp/track` с телом, содержащим `"event": "push_clicked"` и полями `moduleId` и `lessonId`.

**Шаги — проверка на мобильном viewport (375px):**

16. В DevTools установи viewport 375×667 (iPhone SE) и повтори шаги 2–4.
17. Экран урока должен помещаться на экран без горизонтальной прокрутки.
18. Все кнопки должны быть кликабельны (тапабельны) — не обрезаны, высота кнопок не менее 44px.

**Ожидаемое поведение:** при открытии с валидными параметрами — мгновенный переход на урок без флэша дашборда; кнопка назад ведёт на главный экран; невалидные параметры — тихий fallback на Dashboard; событие `push_clicked` фиксируется.

**Если что-то не так:**
- Мини-апп показывает Dashboard вместо урока → F12 → Console, ищи ошибки в `parseDeepLinkParams` или несовпадение id модуля (id чувствителен к регистру: `alphabet`, не `Alphabet`).
- Кнопка «Назад» ведёт в модуль, не на Dashboard → проверь флаг `fromDeepLink: true` в роутере (`App.tsx:142`).
- Событие `push_clicked` не отправляется → проверь Network на 4xx для `/api/miniapp/track`; убедись, что оба параметра moduleId и lessonId присутствуют в URL.
- JS-ошибка при несуществующем модуле → проверь fallback-ветку в `App.tsx:89` (условие `if (mod && lesson)`).
